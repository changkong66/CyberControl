# ADR-0010: Topic 4 Trusted Evidence and Server-Derived Release Boundary

**Status:** Accepted

## Context

Topic 4 C1-C12 is accepted on the protected backend baseline, but its first
runtime integration leaves three delivery boundaries too permissive for a browser
client. C11 has no trusted path for importing the local build evidence consumed by
its fail-closed handler. C12 v1 accepts caller-supplied Candidate, report, and
authorization documents that can be internally self-consistent without proving
that they are the immutable records persisted by Topic 3 and Topic 4. The existing
human-review service is not exposed through a tenant-scoped API.

The accepted migrations `0001` through `0009` and all existing v1 wire fields are
immutable. Security remediation therefore has to be additive and versioned.

## Decision

- C11 gains a trusted evidence-import boundary backed by the existing C11 tables,
  Artifact Store, audit chain, transactional Outbox, and FORCE RLS policies. The
  service loads the authoritative C6 `CodeArtifactV1` from the persisted C6 module
  result; callers cannot select or replace a Tenant, Candidate, Claim, or code
  artifact identity.
- This phase imports CycloneDX JSON, vulnerability observations, and local build
  provenance from an allowlisted deterministic builder. It does not create a
  Python or MATLAB build farm and does not call an external scanner or embedding
  service.
- `ComplianceEvidenceImportCommandV1` and `ComplianceEvidencePackageV1` are
  immutable internal records generated by the server. The HTTP request contains
  only the verification and Claim identifiers plus untrusted evidence documents.
- C12 gains additive `ReleaseDerivationCommandV2` and
  `PublicationCommitCommandV2` records and server-derived v2 endpoints. A client
  may request a release mode, a subset of server-allowed blocks, and a TTL of at
  most 300 seconds. It cannot submit a Candidate, report, Tenant, authorization
  identity, or authoritative SHA.
- C12 v1 contracts and routes remain available for compatibility and are marked
  deprecated. Before issuing or consuming a v1 authorization, the runtime reloads
  the persisted latest report and Topic 3 Candidate and requires exact field and
  SHA equality. A self-consistent caller-forged document is rejected.
- A successful PostgreSQL publication appends the C1 `RELEASED` state snapshot in
  the same SERIALIZABLE transaction as authorization consumption, PENDING and
  COMMITTED publication snapshots, audit, Outbox, and public stream-event writes.
  A replay may return the same committed result only when the immutable
  authorization consumption already exists and the verification is `RELEASED`.
- Existing human-review contracts remain unchanged. The HTTP API materializes the
  immutable decision from the authenticated subject and tenant, then delegates to
  `VerificationService.submit_human_review`. The frozen reject outcome is
  `ReviewDecision.BLOCK`; user interfaces may label it "Reject" but must send the
  existing `BLOCK` value.
- No migration from `0001` through `0009` is edited and no new database migration
  is introduced by this decision.

## Consequences

Frontend code can use C12 v2 without possessing or reconstructing authoritative
backend records. C11 remains local-first and fail-closed when evidence is absent,
expired, cross-tenant, built by an unapproved toolchain, or hash-invalid. Legacy
integrations keep compiling but receive deprecation metadata and stricter
persisted-fact validation. Publication replay and rollback tests must include the
new `RELEASED` state snapshot as part of the atomic boundary.
